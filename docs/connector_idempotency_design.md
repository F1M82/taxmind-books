# Connector Idempotency Layer — Design Session

**Date:** 2026-06-14
**Status:** DESIGN ONLY — no code, no commits, no tests run.
**Arc:** Connector-side idempotency cache, taken as its own track *before*
re-evaluating Direction B (Redis pub/sub fan-out) for BUG-Books-003. The
BUG-003 design session identified the absence of this layer as **BLOCKING-1**:
safe automatic retry of `post_voucher` is impossible without connector-side
dedup, because only the connector knows whether Tally already accepted a given
key.
**Scope owner:** lead engineer (implementation-level architectural authority;
escalates only business/product/security/compliance/UX trade-offs).

---

## Audit summary (what the protocol requires vs what exists)

| Capability | Protocol requires | Connector code today |
|---|---|---|
| Read `idempotency_key` from `command` payload | Yes (`PROTOCOL §command`, line 326) | **No** — `dispatch_command` never reads it |
| `(command, idempotency_key) → result` SQLite cache, 24h | Yes (line 326, 457, 840) | **None** — no SQLite anywhere in connector runtime |
| Replay returns cached result without re-posting | Yes (line 457, 822, test case 6 line 865) | **Not implemented** |
| `post_voucher` idempotent on key | Yes (line 457) | **No** — `_handle_post_voucher` calls Tally directly every time |
| Backend emits `idempotency_key` | Yes (line 765) | **YES** — `voucher_dispatcher.py:373` emits `str(voucher_id)` |
| `outbound_queue` (offline buffering) | Yes (lines 801-822) | **None** — separate, larger feature; **out of scope** for this arc |

**Net:** the emission half exists; the entire consume/dedup half is missing.
This arc builds the consume half. The `outbound_queue` (offline drain) is a
distinct feature the protocol also promises but which is *not* required for
retry-safety and is explicitly excluded here (see "What NOT to do").

# Task 1 — Protocol specification (verbatim extracts)

**Message envelope, on `request_id` (line 121):**
> `request_id`: UUID v4 generated by the sender. Used for correlating
> request/response pairs **and for idempotency.**

**`command` message, `idempotency_key` field (lines 319, 324-328):**
> ```json
> "idempotency_key": "<uuid>"
> ```
> `company_id` is included so the connector can verify the command targets the
> company it registered for. Mismatch → connector replies with `command_result`
> status=error, code=`company_mismatch`, and logs locally.
>
> `idempotency_key` is the connector-side dedup key. **The connector maintains a
> local SQLite cache of `(command, idempotency_key) → result` for 24 hours. A
> repeat command with the same key returns the cached result without
> re-executing against Tally.**
>
> `timeout_seconds` is how long the connector waits for Tally to respond.
> Default 30. Bulk operations use higher values.

**`post_voucher` command (line 457):**
> The post is idempotent on `idempotency_key`. **If the connector's local cache
> shows the same key was previously posted with success, it returns the cached
> result without re-posting.** This handles the case where the backend retries
> the WS command after a network drop, but Tally already accepted the post.

**`post_voucher` args (line 417):** carries `"voucher_id": "<uuid>"` in args, in
addition to the envelope's `idempotency_key`.

**Connector-side outbound queue (lines 805-822):**
> ```sqlite
> CREATE TABLE outbound_queue (
>     id TEXT PRIMARY KEY,
>     request_id TEXT NOT NULL,
>     command TEXT NOT NULL,
>     args TEXT NOT NULL,             -- JSON
>     idempotency_key TEXT NOT NULL,
>     enqueued_at TIMESTAMP NOT NULL,
>     attempts INTEGER NOT NULL DEFAULT 0,
>     last_error TEXT
> );
> CREATE UNIQUE INDEX idx_outbound_idem ON outbound_queue (idempotency_key);
> ```
> A background thread in the connector polls Tally's reachability every 5
> seconds. When Tally returns, the connector drains the queue in FIFO order...
> The backend Celery task for those vouchers will have already failed and
> retried; **when the next retry arrives, the connector's idempotency cache
> prevents double-posting.**

**Security property 4 (line 840):**
> **Idempotency at both layers.** Backend deduplicates HTTP requests
> (`IDEMPOTENCY.md`); **connector deduplicates Tally posts (local SQLite
> cache).** Network retries between the two layers don't double-post.

**Validation test case 6 (line 865):**
> POST the same voucher twice with the same Idempotency-Key. **Verify only one
> Tally post happens** (check Tally voucher list).

**Command-specific idempotency already in spec (lines 512, 539):**
> - `approve_optional_voucher`: `voucher_already_regular` — already non-Optional
>   (**idempotent: returns success**)
> - `reject_optional_voucher`: `tally_voucher_not_found` — already gone
>   (**idempotent: returns success with `deleted=true`**)

**SPEC NOTE (flagged, not a STOP):** the protocol describes **two** SQLite
structures keyed on `idempotency_key` — the *dedup cache* (prose only, line 326)
and the *outbound_queue* (concrete DDL, line 806). They are distinct: the cache
prevents re-execution; the queue buffers commands while Tally is offline. The
brief's requested schema (status, result_payload, request hash, timestamps) maps
to the **cache**, not the queue. The cache's schema is specified only in prose,
so this design defines it concretely below. No internal contradiction — but the
two must not be conflated, and the queue is out of scope.

# Task 2 — Connector code audit (what implements / fails to implement)

**`connector/connector/message_handlers.py`** — does any handler read
`idempotency_key`?
- **No.** `dispatch_command` (lines 153-225) reads `company_id`, `command`,
  `args` — never `payload["idempotency_key"]`. `_handle_post_voucher` (73-77)
  is `return await tally.post_voucher(_voucher_from_args(args))` with no cache
  lookup, no key. Every dispatch executes against Tally unconditionally.

**`connector/connector/tally_client.py`** — does `post_voucher` accept/use an
idempotency key?
- **No.** `post_voucher(self, voucher: VoucherInput)` (524) takes only the
  voucher. `VoucherInput` (127-147) has no idempotency field. The Create
  envelope (`_build_voucher_xml`, 807-839) sets **no REMOTEID** on the
  `<VOUCHER ACTION="Create">` — so Tally is given no external durable key.
- **Relevant gap:** `post_voucher` returns `"tally_voucher_guid": None` (547) —
  Layer C (durable Tally GUID via REMOTEID / `<LASTVCHID>`) is deferred per
  BUG-004. Consequence for this design: a cached result will carry
  `guid=None`; the cache can prevent re-posting but cannot later supply a GUID.

**`connector/connector/ws_client.py`** — does the WS envelope carry idempotency
context?
- The envelope carries `request_id`; `_handle_command` (280-308) forwards the
  **entire** `env["payload"]` (which *does* contain `idempotency_key`) into
  `dispatch_command`. So the key is **present at the dispatch boundary** — it is
  simply never consumed. No code change to `ws_client` is required; the work is
  inside the dispatch/handler layer.

**Any SQLite database file referenced anywhere in connector code?**
- **No.** A repo-wide search for `sqlite|aiosqlite|.db|APPDATA|data_dir` over
  `connector/` returns only build tooling (`installer/build_exe.py`) and test
  fixtures. No runtime persistence of any kind.

**Any outbound queue / request cache / result cache structure?**
- **None.** No queue, no cache, no on-disk state. The connector is currently
  fully stateless across restarts.

**Config audit (`connector/connector/config.py`):** `ConnectorSettings` has
**no data-directory setting**. There is nowhere a SQLite path is configured.
This design introduces one (`CONNECTOR_DATA_DIR`, default `%APPDATA%\TaxMindBooks`
— the same root the protocol/CLAUDE.md already use for `token.dat`).

# Task 3 — Backend-side emission audit

**Does the backend include `idempotency_key` when dispatching `post_voucher`?**
- **YES.** `voucher_dispatcher.dispatch_voucher_to_tally` calls
  `registry.send_command(..., idempotency_key=str(voucher_id))`
  (`voucher_dispatcher.py:373`). The registry's `send_command`
  (`connector_registry.py:138-139`) places it into the command payload.
- **The emitted key is the voucher UUID.** Combined with `command="post_voucher"`,
  the cache key `(command, idempotency_key)` is effectively *the voucher* — which
  is exactly the dedup granularity we want (one Tally post per voucher).

**Consequence:** the backend needs **no change to emission.** This collapses the
arc to the connector consume-half plus (optional) traceability. The brief
anticipated this ("If the backend already emits keys, that simplifies the
work") — it does, materially.

# Task 4 — Tally-native idempotency survey

**Does Tally have native idempotent imports?** Effectively **no**, for the
Create path we use:

- **VOUCHERNUMBER collision:** Tally rejects a duplicate explicit
  `<VOUCHERNUMBER>` within a voucher type (`voucher_number_collision`, protocol
  line 465). But Phase 0 commonly posts with auto-numbering (number may be
  null), so this provides **no** dedup in the common case.
- **REMOTEID:** Tally uses `REMOTEID` as an external durable key for `Alter` /
  `Delete` (`tally_client.py:857, 878`). The **Create** envelope sets no
  REMOTEID (`_build_voucher_xml`). Whether a `Create` with an explicit REMOTEID
  is rejected/no-op'd by Tally on duplicate is **unverified** (needs a live
  Tally). This is the single highest-value research lever — see Open Questions.
- **GUID round-trip:** `post_voucher` returns `guid=None` today (Layer C
  deferred). So there is currently no durable Tally key returned that we could
  use to look a voucher up after the fact.
- **`<LASTVCHID>`** is parsed into `ImportResponse.last_vch_id`
  (`tally_client.py:309`) but is **not** propagated into the post_voucher result
  — a latent hook that Layer C could use.

**Conclusion:** with the current Create envelope, Tally provides **no reliable
native idempotency**. Dedup must be **100% connector-side**, exactly as the
protocol's security-property-4 design intends. (REMOTEID-on-Create could later
add a Tally-native safety net AND deliver Layer C's durable GUID — flagged as
research, not assumed.)

# Task 5 — Design: connector-side idempotency cache

## Guiding invariant

**Never execute a mutating Tally command unless the connector has first durably
recorded its intent.** The cache write is a *precondition* for the Tally call,
not a post-hoc record. This is what shrinks the duplicate-post window from "every
retry" (today) to "only a crash inside one narrow interval" (see Failure Modes).

## State model — SQLite schema

One table, keyed exactly as the protocol specifies, `(command, idempotency_key)`:

```sql
PRAGMA journal_mode = WAL;          -- crash-safe, single-writer-friendly
PRAGMA synchronous  = FULL;          -- durability over speed; volume is tiny
PRAGMA user_version  = 1;            -- migration ladder anchor

CREATE TABLE IF NOT EXISTS command_idempotency (
    command              TEXT NOT NULL,            -- e.g. 'post_voucher'
    idempotency_key      TEXT NOT NULL,            -- backend's voucher UUID
    request_payload_hash TEXT NOT NULL,            -- sha256 of canonical args JSON
    status               TEXT NOT NULL,            -- 'in_flight' | 'completed' | 'failed'
    result_payload       TEXT,                      -- JSON command_result payload (NULL while in_flight)
    created_at           TEXT NOT NULL,            -- ISO-8601 UTC (intent recorded)
    updated_at           TEXT NOT NULL,            -- ISO-8601 UTC (last write: intent or settle)
    PRIMARY KEY (command, idempotency_key)
);

CREATE INDEX IF NOT EXISTS idx_cmd_idem_created
    ON command_idempotency (created_at);       -- supports TTL sweep
```

> **Shipped naming (per DECISION 3):** the design's `request_hash` /
> `completed_at` shipped as **`request_payload_hash`** / **`updated_at`**
> (more self-documenting). `updated_at` advances on every write — both the
> in-flight insert and the terminal settle — not only on completion. See the
> "Shipped implementation" section at the end of this doc.

Notes:
- `request_payload_hash` is defensive: if the *same* key arrives with *different*
  args, that is a backend bug (keys must be stable per logical operation). The
  design returns an error rather than silently serving a stale result for a
  different payload. **(Not yet implemented — see Shipped implementation: the
  hash is stored but the mismatch → `idempotency_key_conflict` check is
  deferred.)**
- `status` is a small, explicit lifecycle (below). Only `completed` and `failed`
  are terminal/cacheable; `in_flight` is the crash-detection sentinel.
- Only **mutating** commands are cached: `post_voucher` (and later
  `approve_optional_voucher` / `reject_optional_voucher`). Read commands
  (`ping`, `sync_masters`, `get_*`) are **not** cached — they are naturally
  idempotent and re-reading is harmless and desirable (fresh data).

## Lifecycle (per mutating command)

```
command arrives (command, idempotency_key, args)
  │  compute request_hash = sha256(canonical_json(args))
  ▼
lookup (command, idempotency_key)
  ├─ COMPLETED  → if request_hash matches: return cached result_payload (NO Tally call)
  │                if mismatch:            return error 'idempotency_key_conflict' (no Tally)
  ├─ FAILED     → return cached failure result_payload (NO Tally call; re-post would re-reject)
  ├─ IN_FLIGHT  → indeterminate prior attempt (concurrent, or crashed):
  │                see Failure Modes §"crash window" — default: retryable
  │                'idempotent_in_flight' (NO Tally call)
  └─ ABSENT     → first execution:
                    1. INSERT row status='in_flight', request_hash, created_at   (durable)
                    2. await handler(tally, args)  → Tally Create
                    3a. success      → UPDATE status='completed', result_payload, completed_at
                    3b. non-retryable→ UPDATE status='failed',    result_payload, completed_at
                    3c. retryable    → DELETE the in_flight row (let backend retry fresh)
                    4. return result
```

The lookup→insert path runs inside a single SQLite transaction guarded by an
`asyncio.Lock` so two concurrent commands with the same key cannot both reach
step 1 (the loser sees `in_flight` and backs off).

## Retry semantics

- **Common case — replay of a completed post** (network drop, WS timeout, worker
  redelivery, Celery autoretry): lookup hits `completed` → cached result
  returned, **Tally never touched.** This is the protocol's line-457 requirement
  and the property that makes BUG-003 retries safe.
- **Replay of a non-retryable failure** (e.g. `tally_validation_failed`): returns
  the cached failure deterministically; re-posting identical args would only
  re-reject. Saves a Tally round-trip and keeps the backend's audit trail
  consistent across retries.
- **Retryable failure** (`tally_unreachable`, `tally_timeout`, ambiguous): the
  in_flight row is **deleted**, so the next attempt genuinely re-executes against
  Tally. We do *not* cache transient failures.

## Cleanup policy (TTL + stuck in-flight)

- **Completed/failed rows:** retained 24h (protocol line 326), then swept. A
  lazy sweep on connector startup + a periodic sweep (piggy-backed on the
  existing heartbeat loop, every N heartbeats) deletes
  `status IN ('completed','failed') AND created_at < now-24h`. 24h comfortably
  exceeds the backend's 5-attempt exponential retry window (≈31 min) and any
  realistic WS reconnect gap.
- **Stuck `in_flight` rows** (crash sentinels): never auto-deleted by the TTL
  sweep (they have no safe expiry — see crash window). They are resolved only by
  the in_flight handling path on the next replay, or by operator reconciliation.
  A stale in_flight older than `2 × timeout_seconds` is *reported* (logged +
  surfaced via a connector `error` message) but **not** silently cleared, so
  duplicates are never risked by a timer.

## Failure modes

1. **SQLite write fails on the in_flight INSERT** (disk full, locked, corrupt):
   per the guiding invariant, **do not post to Tally.** Return retryable
   `idempotency_store_unavailable`; the backend retries later. WAL mode + a short
   `busy_timeout` + bounded retry on `SQLITE_BUSY` handle transient locks. Trade:
   if the store is truly broken, posting halts — strictly safer than unguarded
   double-posts.
2. **Connector crashes AFTER Tally Create but BEFORE the `completed` UPDATE — the
   crash window.** On restart, the replay finds `in_flight`. The connector
   **cannot know** whether Tally accepted the post (no REMOTEID set on Create,
   `guid=None`, so no reliable Tally lookup key exists today).
   - **Default resolution (recommended, keeps arc in budget):** treat stale
     in_flight as **indeterminate** — return retryable `idempotent_in_flight`
     and **never auto-re-post.** Outcome: in the rare crash-window case the
     voucher may be left needing manual reconciliation, but the connector
     **never double-posts.** This honours financial-data correctness (a missing
     post is detectable and fixable; a silent duplicate is corrosive).
   - **Window size:** the interval between Tally returning 200 and one SQLite
     UPDATE committing — milliseconds. With `synchronous=FULL` WAL the UPDATE is
     durable on commit. The exposure is genuinely small but **not zero**.
   - **Full closure requires more than this arc** (see Open Questions OPEN-A):
     either REMOTEID-on-Create (Tally-native idempotency) or a Tally
     reconciliation lookup — both depend on the deferred Layer C GUID work and
     live-Tally verification.
3. **Crash between in_flight INSERT and the Tally call** (Tally never touched):
   restart replay finds in_flight → indeterminate → backend retries. Here
   re-posting is actually *safe* (Tally never saw it), but we can't distinguish
   this from case 2, so the safe-by-default path treats both as indeterminate.
   This is the cost of the conservative default: a never-posted voucher in this
   sub-case also needs a nudge. Acceptable; rare; never duplicates.
4. **Two commands, same key, concurrent** (e.g. backend double-dispatch): the
   `asyncio.Lock` + transactional insert serialises them; the second sees
   in_flight and returns `idempotent_in_flight` (retryable) — by its next retry
   the first has written `completed` and the replay returns the cached result.
5. **Cache corruption / schema drift:** `PRAGMA user_version` gates a migration
   ladder; an unreadable DB is rebuilt from scratch (the cache is a safety
   optimisation, not a system of record — losing it degrades to "every retry
   re-posts," i.e. today's behaviour, which the backend's other guards already
   tolerate). Rebuild-on-corruption is logged loudly.

## Schema migration / first-run creation

- On connector startup, resolve the data dir (`CONNECTOR_DATA_DIR`, default
  `%APPDATA%\TaxMindBooks` on Windows; `~/.local/share/TaxMindBooks` elsewhere),
  `mkdir -p`, open `connector.db`.
- Run `PRAGMA journal_mode=WAL`, `CREATE TABLE IF NOT EXISTS ...`, then a tiny
  `user_version`-based migration ladder (`if user_version < N: apply step N`).
  **No Alembic** — that is backend-grade machinery; the connector needs only a
  handful of idempotent DDL steps. Creation is safe to run on every boot.
- The store init is wired in `connector/main.py` startup, before the WS client's
  `run_forever`, and the opened store handle is passed to the dispatch layer.

## Backend-side changes

- **Emission:** none. `idempotency_key=str(voucher_id)` is already sent
  (`voucher_dispatcher.py:373`).
- **Traceability:** the brief asks about persisting the key in the voucher row.
  Today **key ≡ voucher_id**, so a dedicated column would store derivable data
  and require a model + Alembic migration in `backend/app/models` — *outside*
  the connector arc and a needless schema change. **Recommendation: do not add a
  column now.** Document the invariant "connector idempotency_key for
  post_voucher = voucher_id." Add a `tally_idempotency_key` column only if/when
  the key is ever decoupled from `voucher_id` (not foreseen).
- **Retry posture:** unchanged — but it becomes *correct*. Once the connector
  dedups, the backend's existing `autoretry_for=(ConnectorOffline,
  CommandTimeout, TallyRetryableEnvelope)` and `task_acks_late=True` stop being a
  duplicate-post hazard. **This is the BUG-003 BLOCKING-1 unblock**, achieved
  with zero backend code change.

## Test strategy

All connector-side, deterministic, against a **temp-file or `:memory:` SQLite**
(no real Tally, no real APPDATA, no real crash):

- **Cache miss → execute → store:** dispatch with a mocked `TallyClient`; assert
  handler called once, row written `completed`, result returned.
- **Cache hit (completed) → no re-execute:** pre-seed a `completed` row; dispatch
  the same key; **assert `TallyClient.post_voucher` is NOT called** and the
  cached result is returned. *(This is protocol test case 6, deterministic.)*
- **Cache hit (failed non-retryable) → cached failure, no Tally call.**
- **Retryable failure → no terminal row:** make the mock raise
  `TallyUnreachable`; assert the in_flight row is deleted and a second dispatch
  re-invokes Tally.
- **Key reuse with different args → `idempotency_key_conflict`,** no stale
  result served, no Tally call.
- **Replay-after-crash (the headline deterministic test):** insert an
  `in_flight` row directly (simulating a process that died before the terminal
  UPDATE), then dispatch the same command. **Assert Tally is NOT re-posted** and
  the indeterminate/retryable signal is returned. This reproduces "connector
  crashed between Tally post and persistence" without any real crash.
- **Two-boot replay:** open store → post (mock) → close store → re-open store
  (simulated restart) → replay → assert cached `completed` result returned and
  Tally not called. Proves cross-restart durability.
- **SQLite write failure → no Tally call:** inject a store whose insert raises;
  assert `idempotency_store_unavailable` (retryable) and `post_voucher` never
  invoked (the guiding invariant).
- **Concurrent same-key:** two `asyncio.gather`-ed dispatches; assert exactly one
  Tally call and the other gets `idempotent_in_flight`.
- **Migration idempotency:** run store init twice; no error; `user_version`
  correct.
- **TTL sweep:** seed an old `completed` row + a fresh one; run sweep; assert
  only the stale one is deleted and `in_flight` rows are untouched.

**Integration:** `dispatch_command` end-to-end (real temp SQLite + faked
`TallyClient`) exercised through the same boundary `ws_client._handle_command`
uses, confirming the key flows from envelope → dispatch → cache.

**NOT tested (acceptable deferrals):** real TallyPrime; real OS-level crash
(`SIGKILL`) — simulated via direct in_flight rows; real `%APPDATA%` path — use
tmp dirs; REMOTEID-on-Create Tally behaviour — requires live Tally (Open
Questions).

# Decision summary

| # | Decision | One-line justification |
|---|---|---|
| D1 | Single `command_idempotency` table keyed `(command, idempotency_key)` | Exactly the protocol's stated key (line 326); granularity = one Tally post per voucher. |
| D2 | Cache only **mutating** commands (`post_voucher` + later approve/reject) | Reads are naturally idempotent; caching them would serve stale Tally data. |
| D3 | **Write intent (in_flight) before the Tally call** | Shrinks the duplicate window from "every retry" to "one narrow crash interval"; enables crash detection. |
| D4 | Cache terminal **success** and **non-retryable failure**; delete on **retryable** | Replays of settled outcomes are deterministic; transient failures must genuinely re-execute. |
| D5 | `request_hash` (sha256 of canonical args) stored per row | Detects key-reuse-with-different-args (a backend bug) instead of serving a wrong cached result. |
| D6 | Stale `in_flight` → **indeterminate, never auto-re-post** (safe-by-default) | A missing post is detectable/fixable; a silent duplicate corrupts financial data. |
| D7 | **stdlib `sqlite3`, WAL, `synchronous=FULL`, `asyncio.Lock` around writes** | No new dependency; single-process connector; durability over throughput at trivial volume. |
| D8 | `user_version` migration ladder, **no Alembic**; `CREATE TABLE IF NOT EXISTS` on boot | Connector needs a few idempotent DDL steps, not backend-grade migration tooling. |
| D9 | New `CONNECTOR_DATA_DIR` setting, default `%APPDATA%\TaxMindBooks` | The protocol/CLAUDE.md already root connector state there (`token.dat`); no config home exists today. |
| D10 | **No backend code change** (emission already exists; no traceability column) | Key ≡ voucher_id today; a column would store derivable data and force a model migration outside scope. |
| D11 | `outbound_queue` (offline drain) **excluded** from this arc | It is a separate feature; not required for retry-safety; conflating it blows scope. |

# Open questions surfaced (guardrail items)

**OPEN-A — Closing the crash window fully needs work beyond this arc
(research lever).** The safe-by-default (D6) never double-posts but can leave a
rare voucher needing manual reconciliation. *Full* closure requires Tally-native
or Tally-lookup idempotency, which depends on **REMOTEID-on-Create** behaviour
that is **unverified** (does TallyPrime reject/no-op a duplicate REMOTEID on
`ACTION="Create"`?). That research also overlaps **BUG-004 Layer C** (durable
GUID). *Lead-engineer decision:* accept the narrow residual risk for this arc
(recommended), or fold REMOTEID/Layer C research in (+1-2 sessions, requires live
Tally). **Not a STOP for the founder** — it is an implementation-depth trade-off
within delegated authority — but it is the one genuine "we don't fully know how
to close this without more research" item, surfaced per the guardrail rather than
papered over.

**OPEN-B — Spec describes two SQLite structures (cache vs outbound_queue).**
Documented in Task 1. Not a contradiction; this arc builds the cache and
explicitly defers the queue. Flagged so the boundary is a conscious decision, not
an omission.

**OPEN-C — Cached `guid=None` (Layer C interaction).** Because `post_voucher`
returns `guid=None` today, cached results carry no GUID. The cache is still
correct for dedup (prevents re-post), but cannot retroactively supply a GUID. If
Layer C lands, the cache schema already has `result_payload` to carry it — no
schema change needed, just a richer cached payload.

**Guardrails explicitly checked and NOT tripped:**
- *Spec internally inconsistent / contradicts code materially?* No material
  contradiction. The cache vs queue distinction (OPEN-B) is a scope boundary, not
  an inconsistency. Code simply doesn't implement the spec yet — which is the
  premise of this arc, not a surprise.
- *Materially affects customer UX?* **No.** The cache is invisible to operators:
  a deduped replay returns the *same* success/failure the operator would have
  seen anyway. No error-message changes, no voucher-status-workflow changes, no
  UI surface touched. (The only new operator-visible signal is the rare
  reconciliation case in OPEN-A, which surfaces through existing
  `tally_post_failed`-style channels, not a new UX.)
- *Pre-existing partial idempotency I didn't know about?* The backend **already
  emits** the key — this *reduces* scope, the opposite of a scope-increasing
  surprise; flagged prominently in Task 3. No connector-side cache exists. No
  hidden partial implementation.

# What NOT to do (rejected ideas / anti-patterns)

- **Do NOT post to Tally before recording intent.** Posting first and caching
  after leaves the full "every retry re-posts" hole open on any crash. The
  in_flight pre-write is the whole point (D3).
- **Do NOT auto-re-post on a stale `in_flight` row.** Without a reliable Tally
  lookup key, re-posting risks a silent duplicate. Safe-by-default is
  indeterminate-and-report (D6), never blind re-execution.
- **Do NOT cache retryable failures.** `tally_unreachable`/`tally_timeout` must
  re-execute; caching them would strand a voucher that a simple retry would post.
- **Do NOT build the `outbound_queue` in this arc.** It is offline-buffering, a
  distinct and larger feature; the dedup cache is sufficient for retry-safety
  (D11).
- **Do NOT add Alembic / heavy migration tooling to the connector.** A
  `user_version` ladder with `CREATE TABLE IF NOT EXISTS` is enough (D8).
- **Do NOT add a `tally_idempotency_key` column to the voucher model now.** The
  key is derivable (≡ voucher_id); a column means a needless backend migration
  outside this arc (D10).
- **Do NOT treat the cache as a system of record.** It is a safety optimisation;
  losing/corrupting it degrades to today's behaviour, so rebuild-on-corruption is
  acceptable and must not block startup (D-failure-mode-5).
- **Do NOT change backend retry semantics.** They are already correct *once* the
  connector dedups; touching them is unnecessary and risks regressing the
  BUG-005-validated EAGER path.
- **Do NOT assume REMOTEID-on-Create works.** It is an unverified research lever
  (OPEN-A), not a design assumption. The connector-cache approach must stand on
  its own without it.

## Deviations from the design above (deliberate; documented so the doc and code agree)

1. **Field naming (DECISION 3):** `request_hash` → `request_payload_hash`,
   `completed_at` → `updated_at`; `updated_at` advances on every write.
2. **`in_flight`-on-entry handling — the OPEN-A path.** The Lifecycle pseudocode
   above shows `IN_FLIGHT → return retryable 'idempotent_in_flight' (NO Tally
   call)`. **Shipped behaviour differs by deliberate decision** (session-0
   DECISION 1 / session-2 task 3e): an `in_flight` row on entry is treated as a
   crash sentinel → the connector logs a **WARN and proceeds as first-seen**
   (it DOES call Tally), accepting the bounded duplicate-post risk. This is the
   safe-by-default-for-availability choice: the alternative (never re-posting)
   strands the voucher. Operator reconciliation is the documented fallback.
3. **Hash-mismatch conflict detection NOT implemented.** The Lifecycle
   `COMPLETED → if mismatch: idempotency_key_conflict` branch is deferred.
   `request_payload_hash` is stored for future use; replay currently returns the
   cached result without comparing hashes. (Low risk: the backend's key is the
   voucher UUID, stable per logical operation.)
4. **Settle decision driven off `retryable` (DECISION 5), not exception class.**
   Confirmed alignment with BUG-004 Layer A: `TallyImportRejected`
   (retryable=False) → cached `failed`; `TallyAmbiguousResponse` (retryable=True)
   → `delete_for_retry`, NOT cached as failed. A code comment at the settle site
   records this dependency.
5. **Cleanup TTL = 30 days, not the protocol's 24h.** `CONNECTOR_PROTOCOL.md:326`
   describes a 24-hour cache window; shipped `DEFAULT_TTL_DAYS = 30` (session-3
   decision) for a generous retry window. This is *more* conservative on dedup
   (longer window = safer against late retries) but diverges from the protocol
   text — a future protocol-doc update should reconcile the stated window, or
   the TTL should be lowered to 24h if strict protocol conformance is wanted.

## OPEN-A status — DEFERRED (accepted operational risk)

The crash window (connector dies after Tally accepts a post but before the
`completed` write) remains open, per **session-0 DECISION 1**. Path 1
(safe-by-default: WARN + proceed; rare manual reconciliation) was chosen over
Path 2 (REMOTEID-on-Create / post-crash Tally reconciliation lookup), because
Path 2 depends on the deferred BUG-004 **Layer C** investigation (durable Tally
GUID), which itself needs live-Tally validation that was declined for
environment-economics reasons. OPEN-A naturally closes when Layer C work lands —
the same first-real-user / live-validation event unblocks both. Until then the
duplicate-post exposure is the narrow post→persist interval on an abrupt crash,
and the connector never *silently* double-posts under normal retry (that is what
the cache prevents).
