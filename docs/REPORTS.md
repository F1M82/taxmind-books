# Reports & Analytics — Computation Rules

**Status:** Frozen for Phase 0 (reports) and Phase 1 (analytics).

This document specifies how every report and analytic in TaxMind Books is computed. It is the contract between the API layer and the database. Coder Claude implements queries against this spec; the validation suite tests outputs against TallyPrime's own reports.

The single most important property of these reports: **they must tie out to TallyPrime exactly.** A trial balance generated by TaxMind Books and one generated by Tally on the same date for the same company, with no intermediate manual edits, must produce identical totals. If they don't, our reports are wrong, regardless of what the math says — Tally is the source of truth for the customer.

## Universal rules

These apply to every report and analytic.

### Rule R1 — Optional vouchers are excluded from financial totals

Vouchers with `is_optional_in_tally = true` AND no `approved_to_regular_at` timestamp are excluded from:
- Trial balance
- P&L
- Balance sheet
- Outstanding receivables / payables
- GST liability
- Cash flow
- Top-N analytics
- Aged outstanding

This matches Tally's own behavior: Optional vouchers don't affect financial statements until promoted to Regular.

Optional vouchers ARE included in:
- The pending-approval queue
- The audit log (always)
- A separate "draft view" of reports if the user opts in (a v1.2 concession to power users)

### Rule R2 — Cancelled vouchers are excluded from everything

Vouchers with `status = 'cancelled'` are excluded from all reports and analytics. They remain in the database for audit purposes; they do not contribute to balances.

### Rule R3 — Date range semantics

All reports accept `from_date` and `to_date`. Both are inclusive (`from_date 2026-04-01` and `to_date 2026-04-30` includes both endpoints). Vouchers are filtered by `voucher.date`, not `voucher.created_at`.

If neither is provided, defaults are:
- Trial balance, balance sheet: `as_of_date = today`
- P&L: current financial year (April 1 of current FY → today)
- Outstanding: `as_of_date = today`

### Rule R4 — Financial year semantics

Indian financial year runs April 1 → March 31. The current FY for any date `d`:
- If `d.month >= 4`: FY = `(d.year, d.year+1)` — e.g., 8-May-2026 → FY 2026-27, starts 1-Apr-2026
- If `d.month < 4`: FY = `(d.year-1, d.year)` — e.g., 15-Feb-2027 → FY 2026-27, starts 1-Apr-2026

The company's `financial_year_start` column always stores `April 1 of FY-start year`.

### Rule R5 — Money formatting

All money in API responses is `Decimal` serialized as string per `MONEY.md`. Display formatting (Indian grouping, ₹ symbol) happens on the client. API response example:

```json
{ "value": "1500.00", "value_dr_cr": "Dr" }
```

The API does NOT pre-format with ₹ or commas. The mobile/web client uses `formatINR()` from `mobile/src/utils/money.ts`.

### Rule R6 — Decimal precision

All sums use Python `Decimal` arithmetic per `MONEY.md`. Quantize to 2 decimal places only at the response boundary (and at storage), never during intermediate computation. Banker's rounding (`ROUND_HALF_EVEN`).

### Rule R7 — Caching

Reports are computed on demand for v1. No caching layer. Rationale: a typical MSME has fewer than 10,000 vouchers per year; the trial balance query is fast enough on indexed Postgres without a cache. Premature caching adds an invalidation problem we don't yet need.

If query latency exceeds 2 seconds at 95th percentile in production, we revisit. Until then, no cache.

### Rule R8 — Tenant scoping

Every report query is scoped by `company_id` via `get_scoped_session` per `TENANCY.md`. No exception.

### Rule R9 — Authorization

All report and analytic endpoints require an authenticated user with at least `viewer` role on the active company. No role lower than `viewer` exists; if a user has any membership, they can read reports.

## Report definitions

### Trial Balance

```
GET /api/v1/reports/trial-balance?as_of_date=YYYY-MM-DD
```

**Definition:** The closing balance of every active ledger in the company as of `as_of_date`. The sum of all Dr balances equals the sum of all Cr balances.

**Computation:**

For each ledger `L` in the active company:

```
opening_balance_signed = L.opening_balance × (+1 if L.balance_type='Dr' else -1)

period_movement = SUM(
  ledger_entries.amount × (+1 if entry_type='Dr' else -1)
)
WHERE ledger_entries.ledger_id = L.id
  AND voucher.date <= as_of_date
  AND voucher.status = 'posted'
  AND voucher.is_optional_in_tally = false (per R1)

closing_balance_signed = opening_balance_signed + period_movement

if closing_balance_signed >= 0:
    balance, balance_type = abs(closing_balance_signed), 'Dr'
else:
    balance, balance_type = abs(closing_balance_signed), 'Cr'
```

**Response shape:**

```json
{
  "as_of_date": "2026-05-08",
  "company_id": "uuid",
  "ledgers": [
    {
      "ledger_id": "uuid",
      "ledger_name": "Bank Account",
      "group_name": "Bank Accounts",
      "opening_balance": "10000.00",
      "opening_balance_type": "Dr",
      "period_dr": "75000.00",
      "period_cr": "32000.00",
      "closing_balance": "53000.00",
      "closing_balance_type": "Dr"
    }
  ],
  "totals": {
    "total_dr": "1250000.00",
    "total_cr": "1250000.00",
    "in_balance": true
  },
  "exclusions": {
    "optional_vouchers_excluded_count": 12,
    "cancelled_vouchers_excluded_count": 3
  }
}
```

**Validation property:** `totals.total_dr == totals.total_cr` always. If not, there is a data corruption bug — the validator raises an alert.

**Test:** the human's validation step runs the same date range in TaxMind Books and in TallyPrime. The two reports must show identical closing balances per ledger. Tolerance: zero (not ±₹1; exact match on Decimal).

### Profit & Loss

```
GET /api/v1/reports/profit-loss?from_date=YYYY-MM-DD&to_date=YYYY-MM-DD
```

**Definition:** Income minus expense for the period. Income = ledgers under "Direct Income" + "Indirect Income" groups (which roll up to "Income"). Expense = ledgers under "Direct Expenses" + "Indirect Expenses" + "Purchase Accounts" groups.

**Computation:**

```
For each ledger L in (groups under Income):
    income_amount = period_movement(L) × (-1 if L.balance_type='Cr' else +1)
    # Income ledgers are Cr by nature; positive movement is Cr (a credit increases income)

For each ledger L in (groups under Expense):
    expense_amount = period_movement(L) × (+1 if L.balance_type='Dr' else -1)

total_income = sum(income_amounts)
total_expense = sum(expense_amounts)
gross_profit_or_loss = total_income - total_expense
```

**Group classification:** the `group_name` field on `ledgers` carries Tally's group name. We map Tally groups to P&L vs Balance Sheet vs neither, per a static table in code (`backend/app/services/reporting/tally_groups.py`):

```python
INCOME_GROUPS = {"Direct Incomes", "Indirect Incomes", "Sales Accounts"}
EXPENSE_GROUPS = {"Direct Expenses", "Indirect Expenses", "Purchase Accounts"}
ASSET_GROUPS = {"Bank Accounts", "Cash-in-hand", "Sundry Debtors", "Loans & Advances (Asset)",
                "Current Assets", "Fixed Assets", "Investments", "Stock-in-hand", "Misc. Expenses (Asset)"}
LIABILITY_GROUPS = {"Sundry Creditors", "Current Liabilities", "Loans (Liability)", "Capital Account",
                    "Reserves & Surplus", "Suspense A/c", "Provisions", "Duties & Taxes",
                    "Branch / Divisions", "Secured Loans", "Unsecured Loans"}
```

(This list mirrors Tally's standard chart of accounts. Custom user-created groups inherit their parent's classification.)

**Response shape:**

```json
{
  "from_date": "2026-04-01",
  "to_date": "2026-05-08",
  "income": {
    "ledgers": [
      { "ledger_name": "Sales Accounts", "amount": "1250000.00" },
      { "ledger_name": "Other Income", "amount": "5000.00" }
    ],
    "total": "1255000.00"
  },
  "expense": {
    "ledgers": [
      { "ledger_name": "Purchase Accounts", "amount": "850000.00" },
      { "ledger_name": "Salary", "amount": "150000.00" }
    ],
    "total": "1000000.00"
  },
  "net": {
    "value": "255000.00",
    "type": "profit"
  }
}
```

`net.type` is `"profit"` when income > expense, `"loss"` otherwise. The `net.value` is always positive.

### Balance Sheet

```
GET /api/v1/reports/balance-sheet?as_of_date=YYYY-MM-DD
```

**Definition:** Snapshot of assets, liabilities, and equity as of a date.

**Computation:**

```
For each ledger L in ASSET_GROUPS:
    asset_balance = closing_balance_signed(L, as_of_date) × sign_for_assets
    # Assets are Dr-natured; positive Dr = asset value

For each ledger L in LIABILITY_GROUPS:
    liability_balance = closing_balance_signed(L, as_of_date) × sign_for_liabilities

p_and_l_balance = compute_pnl(start_of_fy(as_of_date), as_of_date)
# P&L for current FY rolls into liability side as Reserves & Surplus equivalent
```

**Response shape:**

```json
{
  "as_of_date": "2026-05-08",
  "assets": {
    "groups": [
      {
        "group_name": "Current Assets",
        "ledgers": [...],
        "total": "850000.00"
      }
    ],
    "total": "1500000.00"
  },
  "liabilities": {
    "groups": [...],
    "total": "1245000.00"
  },
  "current_period_profit_loss": {
    "value": "255000.00",
    "type": "profit"
  },
  "equation": {
    "assets": "1500000.00",
    "liabilities_plus_equity": "1500000.00",
    "in_balance": true
  }
}
```

**Validation property:** `equation.in_balance` always true. If false, reject the response with a 500 error and an alert — the data is inconsistent.

### Outstanding Receivables / Payables

```
GET /api/v1/reports/outstanding?type=receivables&as_of_date=YYYY-MM-DD
GET /api/v1/reports/outstanding?type=payables&as_of_date=YYYY-MM-DD
```

**Definition:** For each party ledger (under Sundry Debtors for receivables, Sundry Creditors for payables), the closing balance as of `as_of_date`.

**Computation:** Same as trial balance, filtered to one group.

**Response shape:**

```json
{
  "type": "receivables",
  "as_of_date": "2026-05-08",
  "items": [
    {
      "ledger_id": "uuid",
      "ledger_name": "Sharma Traders",
      "ledger_gstin": "27BBBBB5678B1Z5",
      "balance": "125000.00",
      "balance_type": "Dr"
    }
  ],
  "total": "1850000.00",
  "total_type": "Dr"

```

For receivables, `balance_type` is typically Dr (party owes us money). For payables, typically Cr (we owe them). Reverse-direction balances (a payable that's currently in advance, etc.) are flagged with `balance_type` as appropriate.

Analytics are Phase 1 features. They are derivative views of the same data.

### GST Liability Summary

```
GET /api/v1/analytics/gst-liability?from_date=YYYY-MM-DD&to_date=YYYY-MM-DD
```

**Definition:** Output GST (collected on sales) minus Input GST (paid on purchases), broken down by tax type.

**Computation:**

```
Output GST (from Sales / Debit Note vouchers):
    cgst_output = SUM(vouchers.cgst) WHERE voucher_type IN ('Sales', 'Debit Note')
    sgst_output = SUM(vouchers.sgst) WHERE same
    igst_output = SUM(vouchers.igst) WHERE same
    cess_output = SUM(vouchers.cess) WHERE same

Input GST (from Purchase / Credit Note vouchers):
    cgst_input = SUM(vouchers.cgst) WHERE voucher_type IN ('Purchase', 'Credit Note')
    sgst_input = SUM(vouchers.sgst) WHERE same
    igst_input = SUM(vouchers.igst) WHERE same
    cess_input = SUM(vouchers.cess) WHERE same

Net liability:
    cgst_payable = max(0, cgst_output - cgst_input)
    sgst_payable = max(0, sgst_output - sgst_input)
    igst_payable = max(0, igst_output - igst_input)
    
    cgst_credit = max(0, cgst_input - cgst_output)
    # etc.
```

**Note:** This is an indicative GST liability, not a filing-grade GSTR-3B computation. Filing requires GSP integration (out of scope for v1). The dashboard tile says "Indicative" to set user expectation.

**Response shape:**

```json
{
  "from_date": "2026-04-01",
  "to_date": "2026-04-30",
  "output": {
    "cgst": "45000.00",
    "sgst": "45000.00",
    "igst": "12000.00",
    "cess": "0.00",
    "total": "102000.00"
  },
  "input": {
    "cgst": "30000.00",
    "sgst": "30000.00",
    "igst": "8000.00",
    "cess": "0.00",
    "total": "68000.00"
  },
  "net_payable": {
    "cgst": "15000.00",
    "sgst": "15000.00",
    "igst": "4000.00",
    "cess": "0.00",
    "total": "34000.00"
  },
  "credit_carryforward": {
    "cgst": "0.00",
    "sgst": "0.00",
    "igst": "0.00",
    "cess": "0.00"
  },
  "disclaimer": "Indicative liability. For filing, refer to GSTR-3B on the GSTN portal."
```

### Aged Outstanding

```
GET /api/v1/analytics/aged-outstanding?type=receivables&as_of_date=YYYY-MM-DD
```

**Definition:** Outstanding receivables/payables bucketed by age of the underlying invoice.

**Computation:**

For each party ledger, walk through unpaid Sales (or Purchase) vouchers in date order. Apply payments (Receipts / Payments against that party) FIFO. The remaining unpaid invoices are the outstanding amount; bucket them by `(as_of_date - voucher.date).days`:

| Bucket | Range |
|---|---|
| Current | 0–30 days |
| 30–60 | 31–60 days |
| 60–90 | 61–90 days |
| 90+ | 91+ days |

**Response shape:**

### Cash Flow (6-month)

```
GET /api/v1/analytics/cash-flow?months=6
```

**Definition:** For each of the last N months (default 6), the total Dr (cash in) and total Cr (cash out) on Bank and Cash ledgers.

**Computation:**

```
For each month M in range [today - N months, today]:
    cash_in[M] = SUM(ledger_entries.amount × (entry_type=='Dr'))
                 WHERE ledger.group_name IN ('Bank Accounts', 'Cash-in-hand')
                   AND voucher.date in M
                   AND voucher.is_optional_in_tally = false
    cash_out[M] = SUM(ledger_entries.amount × (entry_type=='Cr'))
                  WHERE same
    net[M] = cash_in[M] - cash_out[M]
```

**Response shape:**

```json
{
  "months": [
    { "month": "2025-12", "cash_in": "850000.00", "cash_out": "620000.00", "net": "230000.00" },
    { "month": "2026-01", "cash_in": "1100000.00", "cash_out": "750000.00", "net": "350000.00" },
    ...
  ]
}
```

### TDS Payable Summary

```
GET /api/v1/analytics/tds-payable?from_date=YYYY-MM-DD&to_date=YYYY-MM-DD
```

**Definition:** TDS deducted by us (i.e., we owe to government), section-wise.

**Computation:**

```
For each tds_section in DISTINCT(vouchers.tds_section):
    amount = SUM(vouchers.tds_amount)
             WHERE vouchers.tds_section = section
               AND date in range
               AND tds_applicable = true
               AND voucher.is_optional_in_tally = false
```

**Response shape:**

```json
{
  "from_date": "2026-04-01",
  "to_date": "2026-04-30",
  "items": [
    { "section": "194C", "description": "Payment to contractors", "amount": "15000.00", "voucher_count": 8 },
    { "section": "194J", "description": "Professional fees", "amount": "5000.00", "voucher_count": 2 }
  ],
  "total": "20000.00"
}
```

A static lookup table maps section codes to descriptions (`backend/app/services/reporting/tds_sections.py`).

## Data export (DPDP)

```
POST /api/v1/account/data-export
```

**Definition:** User requests a complete dump of their data. The system generates a ZIP containing CSVs (vouchers, ledgers, entries), a JSON of audit logs, and copies of all uploaded files (invoices, statements). A download link is emailed to the user; the link is valid for 7 days.

**Implementation:**
- The request creates a `data_export_requests` row with `status='pending'`.
- A Celery task generates the bundle into S3.
- On completion, an email is sent with a 7-day signed URL.
- Audit log entries: `data_export.requested` and `data_export.completed`.

**Failure modes:**
- Bundle generation fails → email user with apology, retry option, audit log records failure.
- Email delivery fails → user can re-trigger via the API.

**Test cases (validation):**
- Request export → row created, task enqueued
- Task completes → email arrives → link works → ZIP contents are correct (vouchers CSV has all the user's vouchers, no other tenant's data)
- Re-request within 7 days → returns existing link, doesn't regenerate

## Account deletion (DPDP)

```
POST /api/v1/account/deletion-request
DELETE /api/v1/account/deletion-request
```

**Definition:** User requests account deletion. A 30-day grace period begins; during the grace period the user can cancel. After grace, all data is hard-deleted.

**State machine:**

```
none → grace_period (30 days) → completed
                  ↓
            cancelled (back to none)
```

**Implementation:**
- POST: creates `account_deletion_requests` row with `status='grace_period'`, `grace_ends_at = now + 30 days`. Audit log `account.deletion_requested`.
- DELETE: cancels the request. Audit log `account.deletion_cancelled`.
- A daily Celery beat task scans for `grace_period` rows where `grace_ends_at <= now`, switches to `processing`, performs:
  - Generate final data export (R-style above), email to user
  - Hard-delete: vouchers, ledgers, ingestions, drafts, recon data, audit logs of the deleted user (NOT audit logs of other users in the same company — see below)
  - Mark user `is_active=false`, scrub PII (name → "Deleted User", email → "deleted-<id>@deleted.taxmindbooks.in")
- Audit log `account.deletion_completed` (this row is preserved as a final record).

**Edge case: company ownership.** If the deleted user is the sole `owner` of a company, that company's data must also be handled. Options:
1. Block deletion until ownership transferred (preferred — user must re-assign or delete the company first)
2. Auto-transfer to a fallback admin
3. Auto-delete the company

v1.2 implements option 1: deletion is blocked with `409 ownership_transfer_required`. The user must transfer ownership or delete companies first.

**Edge case: company has other users.** If user U is in companies A (sole owner) and B (one of several admins): deleting U requires transferring ownership of A first; B is unaffected (user_companies row deleted, audit logs in B referencing U are kept but `user_id` becomes null and `user_email` snapshot is preserved).

## Cost tracking (admin)

```
GET /api/v1/admin/cost-tracking?company_id=X&month=YYYY-MM
```

**Definition:** Per-company per-month rollup of costs incurred to serve that company:
- LLM tokens (extraction): input + output tokens × Anthropic per-token cost
- Storage: S3 GB-months × per-GB cost
- Compute: estimated based on extraction count × per-extraction compute estimate

Used internally for unit economics. Restricted to `is_superuser=true` users.

**Computation:** the `cost_tracking` table is populated by background workers as events occur:
- Extraction worker writes a row when the LLM call returns
- Storage cron computes total GB monthly
- Compute cron estimates from worker run-time

**Response shape:**

```json
{
  "company_id": "uuid",
  "month": "2026-04",
  "llm_extraction_cost_usd": "12.50",
  "llm_extraction_count": 850,

  "subscription_revenue_inr": "999.00",
  "margin_estimate_inr": "747.00"

```

This data informs pricing decisions; it's not customer-facing.

## Rate limit on AI extraction

The `extraction_quotas` table holds:

| Column | Description |
|---|---|
| company_id | Tenant |
| date | Date (UTC) |
| extraction_count | Count for the day |
| extraction_token_total | Token total for the day |
| soft_limit_alerted | Whether alert has been sent |

Defaults:
- Soft limit: 100 extractions/day OR 500K tokens/day. Beyond this, a notification fires to the admin.
- Hard cap: 500 extractions/day OR 2M tokens/day. Beyond this, new ingestions return `429 extraction_quota_exceeded` until midnight UTC.

Admin can adjust limits per company.

## Validation expectations

The validation report `Section 7.8 — Reports` and `Section 7.9 — Analytics` have these acceptance criteria:

1. Trial balance from TaxMind Books matches Tally's trial balance line-for-line, ledger-for-ledger.
2. P&L matches Tally's P&L net result and the major group totals.
3. Balance sheet matches Tally's balance sheet net result and major group totals.
4. Outstanding matches Tally's "Bills Receivable" / "Bills Payable" reports per party.
5. Optional vouchers do NOT appear in any of the above. Approving an Optional voucher then re-running the report includes the new value.
6. Cancelled vouchers do NOT appear in any of the above.
7. GST liability output values match the sum of GST in the underlying vouchers.
8. Aged outstanding sums match the corresponding party's outstanding total.
9. Cash flow chart values match a manual computation on a sample month.
10. TDS payable values match the sum of TDS in the underlying vouchers, section-wise.

If any of these fail in validation, reports are broken and the phase does not pass. Tally tied-out reports are non-negotiable; the entire trust premise of the product depends on it.
